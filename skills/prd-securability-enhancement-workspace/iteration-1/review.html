Enhanced Feature Specifications\n\n### Feature F-01: Forgot Password\n\n**User story (unchanged)**: A user who does not remember their password can click \"Forgot password?\", enter their email, receive a reset link, and set a new password.\n\n**ASVS Mapping**: V6.2.4, V6.2.12, V6.3.1, V6.3.7, V6.3.8, V6.4.1, V6.4.2, V6.4.3, V6.5.5, V6.6.2, V6.6.3, V7.2.4, V14.1.1.\n\n**Updated Requirements**:\n\n1. **Token generation**: The reset token MUST be generated by a CSPRNG and contain \u00e2\u2030\u00a5128 bits of entropy. Tokens MUST be single-use and bound server-side to a single (user_id, request_id, issued_at) tuple.\n2. **Token lifetime**: Reset links MUST expire \u00e2\u2030\u00a410 minutes after issuance. Expired tokens MUST be rejected with the same response shape as invalid tokens.\n3. **Token storage**: Only a hash of the token (SHA-256 or stronger) is persisted server-side. The plaintext token exists only in the email body and the user's URL.\n4. **Uniform response (no enumeration)**: For any submitted email \u00e2\u20ac\u201d registered, unregistered, malformed, or locked \u00e2\u20ac\u201d the API MUST return HTTP 200 within a bounded time envelope and a generic message (\"If that email is registered you will receive a reset link\"). Response time variance between \"user exists\" and \"user does not exist\" MUST be \u00e2\u2030\u00a450 ms (enforced by deferring email send to an async worker after a constant-time response).\n5. **Rate limiting**: Issuance is rate-limited at three layers:\n - Per email address: \u00e2\u2030\u00a43 reset requests per 15 minutes.\n - Per source IP: \u00e2\u2030\u00a410 reset requests per 15 minutes (with allowance for shared NATs documented).\n - Global: circuit-break SendGrid send-rate above documented threshold.\n6. **No knowledge-based fallback**: There MUST NOT be any \"security question\" / \"memorable date\" challenge anywhere on the reset path.\n7. **MFA preservation**: If the user has 2FA enabled, the reset flow MUST require a successful 2FA challenge (TOTP or recovery code) on the new-password page **before** the password change is committed. Resetting the password alone MUST NOT log the user in or remove the 2FA factor.\n8. **Password quality at reset**: The new password MUST be:\n - \u00e2\u2030\u00a58 characters (15+ recommended in copy);\n - Checked against a top-3000-weak list and a breached-password set (e.g., HIBP k-anonymity API);\n - Accepted as-typed (no truncation, case folding, or composition rules).\n9. **Session hygiene**: On successful reset, the system MUST (a) invalidate **all** existing sessions and refresh tokens for the user, and (b) issue a new session token for the current browser only after the user authenticates with the new password (i.e., reset does not auto-login).\n10. **User notification**: A notification email (\"Your password was reset on {time} from {coarse_location}\") MUST be sent to the on-file address on every successful reset, with a link to a self-service \"this wasn't me\" workflow.\n11. **Logging**: Issue, redeem, expire, and reuse-attempt events MUST be logged as structured security events (see cross-cutting V16 requirements).\n\n**Acceptance Criteria (testable)**:\n\n- AC-01.1: A token issued at T expires by T+10min; redemption at T+10min+1s returns the generic invalid-token response.\n- AC-01.2: A token, once redeemed, cannot be redeemed again; second redemption returns the generic invalid-token response and emits a `recovery.reset.token_reuse_attempt` log event.\n- AC-01.3: For 100 trial submissions split 50/50 between a known email and a known-unregistered email, the 95th-percentile response-time delta is \u00e2\u2030\u00a450 ms and the response body is byte-identical.\n- AC-01.4: Submitting a 4th reset request for the same email within 15 minutes returns a 429 (or the same generic 200 with no email sent) and emits an anti-automation log event.\n- AC-01.5: Attempting to set the new password to \"Password1!\" (in the top-3000 list) is rejected with a clear message; attempting to set a known-breached password is rejected with a distinct, generic \"this password has appeared in a known breach\" message.\n- AC-01.6: For a user with 2FA enabled, completing the reset link without providing a valid TOTP/recovery code does not change the password and does not log the user in.\n- AC-01.7: After a successful reset, all of the user's prior session cookies/refresh tokens fail authorization on the next request (verified across two browsers).\n- AC-01.8: On every successful reset, a \"your password was reset\" email arrives at the on-file address within 60 s.\n\n**Securability Notes**: This is the highest-risk feature in the PRD; account-recovery flows are the dominant takeover vector once login is hardened. The trust boundary is the `/recovery/request` and `/recovery/confirm` endpoints \u00e2\u20ac\u201d strict validation lives there, while interior flow is kept simple (single state machine: REQUESTED \u00e2\u2020\u2019 REDEEMED | EXPIRED). **Confidentiality** drives token-hash-at-rest, no-enumeration response symmetry, and minimal logging of identifying material (log user_id, never the email body or token). **Accountability** drives structured logging of every state transition, tied to user_id and request_id. **Integrity** drives the explicit MFA-preservation rule \u00e2\u20ac\u201d without it, a password reset becomes an MFA bypass. **Resilience** drives async email dispatch (so SendGrid latency cannot widen the timing oracle) and the three-layer rate limit. Avoid hand-rolling token generation or hashing; reuse the platform's auth primitives so the reset path stays modifiable as crypto guidance evolves.\n\n---\n\n### Feature F-02: Account Lockout Recovery\n\n**User story (unchanged)**: After 5 failed login attempts in 10 minutes the account is locked. The user can request an unlock email; clicking the link unlocks the account.\n\n**ASVS Mapping**: V6.1.1, V6.3.1, V6.3.8, V6.4.1, V6.5.5, V6.6.3.\n\n**Updated Requirements**:\n\n1. **Lockout policy (documented)**: The lockout policy is: 5 failed password verifications within a rolling 10-minute window per (user_id) **and** per (source IP \u00c3\u2014 user_id) trigger a soft-lock. Soft-lock duration: 15 minutes auto-expire **or** explicit unlock via email link.\n2. **Counter behaviour**: A successful authentication resets the per-(user_id) counter. Failed attempts against unknown usernames are rate-limited per-IP only and MUST NOT enumerate users (response and timing identical to wrong-password-on-known-user).\n3. **No-enumeration on locked state**: The login response for a locked account MUST be indistinguishable from a generic \"invalid credentials\" response. The \"your account is locked\" UI message is shown **only** after the user clicks \"Get unlock email\" and successfully redeems the unlock link, OR via in-app notification once they re-authenticate. (Trade-off accepted: slightly lower UX clarity on locked accounts, in exchange for no enumeration.)\n4. **Unlock token**: CSPRNG-generated, \u00e2\u2030\u00a5128 bits entropy, single-use, server-side hash-at-rest, lifetime \u00e2\u2030\u00a410 minutes, bound to (user_id, lockout_event_id).\n5. **Rate-limited issuance**: Unlock-email issuance follows the same 3-per-15-minutes per-email and 10-per-15-minutes per-IP envelope as F-01.\n6. **Malicious-lockout protection**: The system MUST NOT permanently lock an account based on failed attempts alone. Auto-expire after 15 minutes is mandatory. Repeated lockout cycles for the same user (>3 in 24h) MUST raise a `recovery.lockout.suspected_targeted_attack` log event for SOC review and MAY apply a longer cooldown, but MUST NOT deny the legitimate user the unlock-email path.\n7. **MFA-aware**: For 2FA users, redeeming the unlock link clears the lockout but does NOT bypass the next 2FA challenge.\n8. **Logging**: Lock, unlock-issue, unlock-redeem, auto-expire, and suspected-attack events all emit structured security log entries.\n\n**Acceptance Criteria (testable)**:\n\n- AC-02.1: After 5 failed password attempts in 10 minutes for a known user, the 6th valid login attempt fails with a generic invalid-credentials response (not a lockout-specific message).\n- AC-02.2: 5 failed attempts against a non-existent username return responses byte-identical (and within 50 ms p95) to 5 failed attempts against a known user with a wrong password.\n- AC-02.3: A lockout auto-expires 15 minutes after the most recent failed attempt, and the user can log in normally without using the unlock link.\n- AC-02.4: An unlock token issued at T cannot be redeemed at T+10min+1s.\n- AC-02.5: An unlock token cannot be redeemed twice; second redemption emits `recovery.unlock.token_reuse_attempt`.\n- AC-02.6: Triggering 4 unlock-email requests for the same email in 15 minutes results in only 3 emails and a logged anti-automation event.\n- AC-02.7: For a 2FA-enabled user, redeeming the unlock link does not log them in until they complete a TOTP or recovery-code challenge.\n\n**Securability Notes**: Lockout is itself an attacker primitive (DoS via known emails) \u00e2\u20ac\u201d so the design must balance brute-force defense (the original intent) against malicious lockout. **Resilience** therefore drives auto-expire as the primary recovery path; the unlock email is a faster opt-in, not a gate. **Authenticity** is preserved by ensuring unlock does not bypass MFA. **Analyzability** is improved by emitting distinct event types for lockout-issued, unlock-redeemed, and auto-expired so SOC can graph attack patterns. The lockout counter belongs in the same trust-boundary module as the login endpoint \u00e2\u20ac\u201d do not let downstream services maintain their own ad-hoc counters.\n\n---\n\n### Feature F-03: Recovery Codes for 2FA Users\n\n**User story (unchanged)**: 2FA users can generate a one-time set of 10 recovery codes from account settings. If the 2FA device is lost, a recovery code can be used in place of TOTP at login. Used codes are invalidated.\n\n**ASVS Mapping**: V6.5.1, V6.5.2, V6.5.3, V6.5.4, V6.5.6, V6.6.3, V14.1.1.\n\n**Updated Requirements**:\n\n1. **Generation**: Codes MUST be generated by a CSPRNG. Each code MUST contain \u00e2\u2030\u00a520 bits of entropy (e.g., 10 alphanumeric chars from a 32-char alphabet \u00e2\u2030\u02c6 50 bits, recommended). Set size: 10 codes.\n2. **Storage**: Codes MUST be hashed with an approved password-hashing algorithm (Argon2id preferred; bcrypt cost \u00e2\u2030\u00a512 acceptable) using a per-code 32-bit (or larger) random salt. Plaintext codes are presented to the user **once** at generation time and never persisted plaintext server-side.\n3. **Display**: Plaintext codes are shown **once** with a \"save these now \u00e2\u20ac\u201d they will not be shown again\" warning, plus a download-as-text-file option. The page MUST set `Cache-Control: no-store` and use `autocomplete=\"off\"` on any field re-displaying them.\n4. **Single-use enforcement**: A redeemed code is marked redeemed atomically (DB row update with a `redeemed_at` timestamp under a transaction or compare-and-set). Concurrent redemption of the same code MUST result in exactly one success.\n5. **Set replacement**: Generating a new set MUST invalidate **all** prior codes atomically before any new code is shown.\n6. **Revocation**: The user can manually revoke their current set (e.g., from a \"Lost my codes\" link) without generating a new one; this leaves the account with TOTP-only recovery and MUST emit a security log event.\n7. **Rate limiting on redemption**: Recovery-code submission at login is rate-limited to 5 attempts per 15 minutes per (user_id + IP). Exceeding this triggers the same lockout flow as F-02.\n8. **No code logging**: Plaintext codes MUST NEVER appear in any log, error message, exception trace, analytics event, or third-party telemetry. Hashed codes MAY appear in audit logs only as a salted hash prefix (first 8 hex chars) for correlation.\n9. **Notification**: Generation, regeneration, and revocation events MUST trigger a notification email to the on-file address.\n10. **Classification**: Recovery code material is classified \"high sensitivity / authenticator\" in the data classification register (V14.1.1).\n\n**Acceptance Criteria (testable)**:\n\n- AC-03.1: Generated codes pass a chi-squared randomness check across a sample of 10,000 codes (no bias detectable at \u00ce\u00b1=0.01).\n- AC-03.2: A code redeemed once cannot be redeemed again, even if submitted within the same TCP request burst (concurrency test with 50 parallel submissions of the same code yields exactly one success).\n- AC-03.3: Generating a new set of codes immediately invalidates all 10 prior codes (verified by attempting to redeem an old code post-generation).\n- AC-03.4: Stored code records contain a hash and salt; database dump inspection reveals no plaintext codes anywhere (codes table, audit log, error log, mail-queue table).\n- AC-03.5: Submitting 6 incorrect codes in 15 minutes triggers lockout per F-02.\n- AC-03.6: Generation, regeneration, and revocation each trigger a notification email and a corresponding `recovery.codes.{generated|regenerated|revoked}` log event.\n- AC-03.7: Inspection of application logs after a full redeem flow shows no occurrence of any plaintext code value.\n\n**Securability Notes**: Recovery codes are bearer authenticators \u00e2\u20ac\u201d they substitute for the second factor entirely. **Confidentiality** therefore dominates: hash-at-rest with proper KDF, never log plaintext, present-once UX. **Integrity** drives atomic single-use enforcement; a TOCTOU race here means the same code authenticates twice. **Authenticity** drives the \"regenerate invalidates prior set\" rule \u00e2\u20ac\u201d without it, a leaked old set remains live. **Modifiability** matters because crypto best practice changes: storing the hash algorithm identifier alongside the hash (e.g., `argon2id$...`) lets you rotate KDFs without a forced re-issuance.\n\n---\n\n### Feature F-04: Recovery via Support Agent\n\n**User story (unchanged)**: If automated recovery fails, the user contacts support. A support agent verifies the user's identity (verbal challenge against on-file data) and triggers a one-time bypass that sends an emergency reset link to the email on file.\n\n**ASVS Mapping**: V6.3.7, V6.4.3, V8.1.1, V8.2.1, V16.3.1, V16.3.2, V16.3.3.\n\n**Updated Requirements**:\n\n1. **Verification script (documented)**: A documented identity-verification script (knowledge factors only the legitimate user would have \u00e2\u20ac\u201d recent order ID + last 4 of billing zip + email-on-file confirmation, never SSN/full DOB) MUST be followed. The script and its acceptance criteria MUST be version-controlled and reviewed quarterly. Knowledge-based questions (\"what was your favorite teacher\") are NOT permitted (V6.4.2).\n2. **Restricted authorization**: The \"trigger emergency reset\" function MUST only be callable by accounts in the `support.recovery_admin` role. Function-level access is enforced server-side; the agent UI MUST NOT be the sole gate.\n3. **No silent MFA bypass**: The emergency action MUST send a reset link to the email on file (NOT to an attacker-supplied address) AND MUST require the user to complete their existing 2FA challenge on the new-password page if 2FA was enabled. If the user has lost both password and 2FA, the agent MUST NOT be able to clear MFA in a single action \u00e2\u20ac\u201d clearing MFA is a separate, logged action that triggers a 24-hour cooling-off period during which the account cannot be logged into until the user confirms the change via a separate email-on-file confirmation.\n4. **Two-channel confirmation for high-risk overrides**: When both password reset AND MFA reset are requested in the same support interaction, the system MUST require a second support agent (or a supervisor role) to approve before the action commits. The approving agent MUST be a different identity than the initiating agent.\n5. **User notification**: The user MUST receive an email (and SMS, if a verified phone is on file) on any support-initiated recovery action, including agent identifier (anonymized to \"Support Agent #ID\") and ticket reference.\n6. **Audit trail**: Every support-initiated recovery event MUST be logged with: timestamp (UTC), agent_id, supervisor_id (if any), ticket_id, action_type, target_user_id, source IP of agent terminal, and outcome. These records MUST be retained \u00e2\u2030\u00a51 year and stored in the immutable security log store (see V16.4).\n7. **Rate / volume limits**: A single agent MUST NOT be able to trigger >10 emergency resets per 24-hour rolling window without supervisor approval.\n8. **No password choosing**: Agents MUST NOT be able to set or view the user's new password (V6.4.6 alignment, even though that requirement is L3 \u00e2\u20ac\u201d applied here as a hardening because of the elevated-privilege nature of the function).\n\n**Acceptance Criteria (testable)**:\n\n- AC-04.1: An agent without the `support.recovery_admin` role attempting to call the emergency-reset endpoint receives 403 and an authorization-failure log event is emitted.\n- AC-04.2: An emergency-reset always dispatches the email to the address currently on file; submitting a different target address in the request body is ignored (server uses on-file address only).\n- AC-04.3: For a 2FA-enabled user, the emergency-reset link does not bypass MFA on the new-password page.\n- AC-04.4: A combined password+MFA reset cannot complete with a single agent's action \u00e2\u20ac\u201d the audit log shows two distinct authenticated agent identities before commit.\n- AC-04.5: Every emergency reset emits a structured log entry with all fields listed in requirement 6, and these entries appear in the SIEM within 5 minutes.\n- AC-04.6: A single agent's 11th emergency reset within 24h is blocked pending supervisor approval; an alert is raised.\n- AC-04.7: The user receives a notification email (and SMS where applicable) within 60 s of any support-initiated action.\n- AC-04.8: The agent UI never displays a \"set new password\" field for the target user (manual code review + functional test).\n\n**Securability Notes**: This is the only flow with a privileged human in the loop, which makes it the highest-trust and highest-risk path simultaneously. **Accountability** is the dominant SSEM concern \u00e2\u20ac\u201d every step needs an attributable, tamper-resistant audit record because the social-engineering attack surface is large. **Authenticity** drives the \"agent cannot redirect the email\" rule and the supervisor co-sign on combined password+MFA resets, which mirrors the FIASSE trust-boundary discipline (the agent is *inside* the system but is still subject to strict validation at this control). **Confidentiality**: the documented script MUST avoid SSN/full DOB to prevent agents from accumulating or being phished out of high-value PII. **Modifiability**: the verification script will need to evolve as fraud patterns shift, so it lives in version control with a change-review process \u00e2\u20ac\u201d not in a wiki page that drifts.\n\n---\n\n## 4. Cross-Cutting Securability Requirements\n\nThese apply to **all** four features and to the recovery surface as a whole.\n\n### 4.1 Anti-Automation & Rate Limiting (ASVS V6.3.1, V2.4.1, V6.6.3)\n\n- A central rate-limiting middleware MUST front all four recovery endpoints (`/recovery/request`, `/recovery/confirm`, `/recovery/unlock-request`, `/recovery/unlock-confirm`, `/recovery/code-redeem`, `/support/emergency-reset`).\n- Default envelopes (per-email, per-IP, global) are configurable without code change.\n- Triggered limits MUST emit `anti_automation.triggered` events with the limit type and identifier.\n- AC-X.1: Load test of 100 req/s sustained against `/recovery/request` from a single IP shows \u00e2\u2030\u00a599% rejected after the configured threshold and the application latency on legitimate endpoints unaffected (>p95 latency degradation <10%).\n\n### 4.2 Authentication-Pathway Consistency (ASVS V6.3.4)\n\n- The four recovery features MUST be enumerated in the authentication architecture document (V6.1.1, V6.3.4) alongside the existing username/password and 2FA login paths.\n- No \"shadow\" recovery path may exist (e.g., legacy admin-only endpoint, staff-only \"impersonate\" feature).
